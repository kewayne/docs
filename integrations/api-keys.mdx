API keys are the primary authentication mechanism for connecting external systems — ERPs, webhook receivers, automation platforms, and custom scripts — to Shiipp Warehouse. Each courier partner gets its own unique key, giving you granular control over which integrations can access which courier's data, and making it easy to revoke access without disrupting unrelated systems.

## How API Keys Work

Every courier partner in Shiipp Warehouse can have one or more active API keys. Keys are passed in the `X-API-KEY` request header on every API call. Access is scoped to **courier-level operations**, meaning a key issued for Courier A cannot read or modify data belonging to Courier B.

<Info>
  API keys grant full read/write access to the courier's data scope. Treat them with the same care as passwords — store them in a secrets manager and never expose them in client-side code or version control.
</Info>

## Generating an API Key

<Steps>
  <Step title="Navigate to the Courier">
    Go to **Couriers** and select the courier partner you want to generate a key for.
  </Step>
  <Step title="Open API Key Settings">
    Click **Settings → API Keys**.
  </Step>
  <Step title="Generate a New Key">
    Click **Generate New Key**. Give the key a descriptive label (e.g., `shopify-webhook`, `erp-sync`) so you can identify its purpose later.
  </Step>
  <Step title="Copy the Key Immediately">
    Copy the full key from the confirmation dialog. **This is the only time the complete key is displayed.** After you close the dialog, only the last four characters are visible in the dashboard.
  </Step>
  <Step title="Store It Securely">
    Save the key in a **password manager** or **secrets vault** (e.g., 1Password, HashiCorp Vault, AWS Secrets Manager) before closing the dialog.
  </Step>
</Steps>

### Using Your Key

Pass the API key in the `X-API-KEY` header with every request:

<CodeGroup>
```bash curl
curl -X GET https://app.shiipp.com/api/packages \
  -H 'X-API-KEY: sk_live_your_key_here'
```

```python Python
import requests

headers = {
    "X-API-KEY": "sk_live_your_key_here"
}

response = requests.get(
    "https://app.shiipp.com/api/packages",
    headers=headers
)

print(response.json())
```

```javascript Node.js
const response = await fetch('https://app.shiipp.com/api/packages', {
  method: 'GET',
  headers: {
    'X-API-KEY': process.env.SHIIPP_API_KEY
  }
});

const data = await response.json();
console.log(data);
```
</CodeGroup>

<Info>
  In the Node.js example above, the key is read from an environment variable (`process.env.SHIIPP_API_KEY`) rather than hard-coded. Follow the same pattern in every environment to keep credentials out of your source code.
</Info>

## Key Rotation

Rotate API keys periodically — or immediately after a suspected exposure — using the zero-downtime process below. Because you generate the new key before revoking the old one, your integrations experience no interruption.

<Steps>
  <Step title="Generate a New Key">
    Follow the [Generating an API Key](#generating-an-api-key) steps above to create a new key for the same courier. Both the old and new keys are active simultaneously during this window.
  </Step>
  <Step title="Update All Integrations">
    Replace the old key with the new key in every integration that uses it — webhooks, ERP connectors, automation scripts, and CI/CD pipelines.
  </Step>
  <Step title="Verify All Integrations">
    Confirm that each updated integration is making successful API calls with the new key before you proceed. Check logs and the Shiipp Warehouse request history to validate.
  </Step>
  <Step title="Revoke the Old Key">
    Once you are confident all integrations are using the new key, return to **Settings → API Keys**, find the old key by its label, and click **Revoke**.
  </Step>
</Steps>

<Warning>
  Revoking a key is **immediate and irreversible**. Any integration still using the revoked key will begin receiving `401 Unauthorized` errors instantly. Always update and verify all integrations **before** revoking the old key.
</Warning>

## Best Practices

<AccordionGroup>
  <Accordion title="One key per integration">
    Issue a separate API key for each integration — one for your webhook receiver, one for your ERP sync, one for your automation platform, and so on. This way, if one key is compromised, you can revoke it without disrupting the others.
  </Accordion>
  <Accordion title="Rotate every 90 days">
    Schedule key rotation at least every **90 days** as a standing security practice, even if no exposure has occurred. Add a recurring calendar reminder or automate rotation through your secrets manager if possible.
  </Accordion>
  <Accordion title="Never log or expose keys in client-side code">
    API keys must never appear in browser JavaScript, mobile app binaries, public repositories, or server logs. Audit your codebase and CI/CD pipeline to ensure keys are injected at runtime from environment variables or a secrets vault.
  </Accordion>
  <Accordion title="Use environment variables">
    Inject keys into your applications through environment variables (e.g., `SHIIPP_API_KEY`) rather than hard-coding them. This keeps credentials out of version control and makes rotation as simple as updating a single variable in your deployment configuration.
  </Accordion>
</AccordionGroup>
