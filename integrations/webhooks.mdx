Webhooks let you push real-time event data from Shiipp Warehouse directly to any external endpoint the moment something changes — a new package arrives, a customer record updates, or a shipment status flips. Instead of polling the API, your systems receive instant, structured payloads so your fulfillment workflows, ERPs, and customer-facing portals stay in sync without delay.

## Overview

Shiipp Warehouse supports both inbound and outbound webhooks, configured per courier partner. Each webhook connection is secured with **AES-256 encryption** in transit, and every payload is signed so your endpoint can verify the source. The system supports three core event types that cover the most critical moments in the package lifecycle.

## Event Types

Subscribe to one or more of the following events when configuring a webhook:

| Event | Trigger |
|---|---|
| `customer_sync` | New customer record added or updated |
| `package_intake` | Package received at warehouse |
| `status_update` | Package status changed |

## Configuring a Webhook

<Steps>
  <Step title="Navigate to Webhook Settings">
    Go to **Couriers**, select the courier partner you want to configure, then open **Settings → Webhooks**.
  </Step>
  <Step title="Enter Your Endpoint URL">
    Paste the full HTTPS URL of the endpoint that will receive webhook deliveries.
  </Step>
  <Step title="Select Events">
    Choose the events you want to subscribe to: `customer_sync`, `package_intake`, `status_update`, or any combination.
  </Step>
  <Step title="Copy the Webhook Secret">
    After saving, copy the generated **webhook secret**. You will use this to verify payload signatures on your server.
  </Step>
  <Step title="Save the Webhook">
    Click **Save**. Shiipp Warehouse will immediately begin delivering events to your endpoint.
  </Step>
</Steps>

## Payload Template Variables

Shiipp Warehouse includes a drag-and-drop **Webhook Payload Designer** that lets you visually map data fields into any JSON structure your downstream system expects. Open it from the webhook configuration screen and drag variables from the left panel into your payload template on the right.

The following variables are available for use in any payload template:

<AccordionGroup>
  <Accordion title="Package & Tracking">
    | Variable | Description |
    |---|---|
    | `{{package_id}}` | Unique package identifier assigned by Shiipp Warehouse |
    | `{{tracking}}` | Carrier-assigned tracking number |
    | `{{status}}` | Current package status (e.g., `received`, `in_transit`, `delivered`) |
    | `{{courier}}` | Name of the courier partner handling the package |
  </Accordion>
  <Accordion title="Weight">
    | Variable | Description |
    |---|---|
    | `{{weight_lbs}}` | Package weight in pounds |
    | `{{weight_kg}}` | Package weight in kilograms |
  </Accordion>
  <Accordion title="Customer">
    | Variable | Description |
    |---|---|
    | `{{customer_name}}` | Customer's full name as recorded in Shiipp Warehouse |
  </Accordion>
</AccordionGroup>

<Tip>
  Use the **Webhook Payload Designer** drag-and-drop builder to visually map variables into your custom JSON structure without writing template code by hand. Changes preview in real time before you save.
</Tip>

## Sample Payload

The following is a representative payload sent on a `package_intake` event using the available template variables:

```json
{
  "event": "package_intake",
  "package_id": "{{package_id}}",
  "tracking": "{{tracking}}",
  "customer": "{{customer_name}}",
  "courier": "{{courier}}",
  "weight": {
    "lbs": "{{weight_lbs}}",
    "kg": "{{weight_kg}}"
  },
  "status": "{{status}}"
}
```

At delivery time, Shiipp Warehouse replaces every `{{variable}}` with the live value from the event record before sending the request to your endpoint.

## Verifying Webhook Signatures

Every webhook delivery includes an `X-Webhook-Signature` header containing an HMAC-SHA256 digest of the raw request body, signed with the **shared secret** you copied during setup. Verify this header on your server before processing any payload to ensure the request genuinely originates from Shiipp Warehouse.

**Verification steps:**

1. Read the raw request body as bytes — do not parse JSON first.
2. Compute `HMAC-SHA256(raw_body, your_webhook_secret)`.
3. Compare the result (hex-encoded) to the value in `X-Webhook-Signature`.
4. Reject any request where the signatures do not match.

<CodeGroup>
```python Python
import hmac
import hashlib

def verify_signature(raw_body: bytes, secret: str, header: str) -> bool:
    expected = hmac.new(
        secret.encode(),
        raw_body,
        hashlib.sha256
    ).hexdigest()
    return hmac.compare_digest(expected, header)
```

```javascript Node.js
const crypto = require('crypto');

function verifySignature(rawBody, secret, header) {
  const expected = crypto
    .createHmac('sha256', secret)
    .update(rawBody)
    .digest('hex');
  return crypto.timingSafeEqual(
    Buffer.from(expected),
    Buffer.from(header)
  );
}
```
</CodeGroup>

<Warning>
  Always use a **constant-time comparison** (e.g., `hmac.compare_digest` in Python, `crypto.timingSafeEqual` in Node.js) when checking signatures. Standard equality checks are vulnerable to timing attacks.
</Warning>

## Retries and Failures

If your endpoint returns a non-`200` status code or does not respond in time, Shiipp Warehouse marks the delivery as failed and automatically retries with exponential back-off. Failed deliveries surface as **Failed Webhook** alerts on the executive dashboard so your team can investigate without digging through logs.

<Warning>
  Webhook endpoints must respond with **HTTP 200** within **30 seconds** or the delivery is marked as failed. Perform any long-running processing asynchronously — acknowledge receipt immediately, then handle the payload in a background job.
</Warning>

<Info>
  The executive dashboard shows the full delivery history for each webhook, including response codes, latency, and retry attempts. Use this view to diagnose intermittent failures before they affect your workflow.
</Info>
