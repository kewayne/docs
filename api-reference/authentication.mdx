Every request to the Shiipp Warehouse REST API must be authenticated with an API key. Authentication is handled through a single request header — no OAuth flows, no token exchanges. You generate a key once inside your Shiipp dashboard, include it with every request, and the API grants access scoped to that courier account.

---

## Passing Your API Key

Include your API key in the `X-API-KEY` header on every request. Requests that omit this header, or supply an invalid key, are rejected immediately before any business logic runs.

```bash
curl https://yourapp.com/api/packages \
  -H 'X-API-KEY: sk_live_your_key_here'
```

The examples below show how to set the header in the most common HTTP clients and languages.

<CodeGroup>

```bash cURL
curl https://yourapp.com/api/packages \
  -H 'X-API-KEY: sk_live_your_key_here'
```

```python Python
import os
import requests

response = requests.get(
    'https://yourapp.com/api/packages',
    headers={
        'X-API-KEY': os.environ['SHIIPP_API_KEY']
    }
)

data = response.json()
```

```javascript Node.js
const response = await fetch('https://yourapp.com/api/packages', {
  headers: {
    'X-API-KEY': process.env.SHIIPP_API_KEY,
  },
});

const data = await response.json();
```

</CodeGroup>

## Where to Find Your Key

API keys are generated and managed inside the Shiipp Warehouse dashboard. Follow the steps below to create your first key.

<Steps>
  <Step title="Open the Couriers section">
    Log in to your Shiipp Warehouse dashboard and navigate to the **Couriers** section in the main sidebar.
  </Step>
  <Step title="Select a courier">
    Click on the courier account you want to generate an API key for. Each courier has its own isolated set of keys.
  </Step>
  <Step title="Open Settings">
    Inside the courier detail page, click the **Settings** tab in the top navigation.
  </Step>
  <Step title="Navigate to API Keys">
    Scroll to the **API Keys** section within Settings.
  </Step>
  <Step title="Generate a new key">
    Click **Generate** to create a new API key. Copy the key immediately and store it in a secure location — Shiipp only displays the full key once.
  </Step>
</Steps>

<Note>
  If you lose a key after closing the generation dialog, you cannot recover it. You must revoke it and generate a new one.
</Note>

## Authentication Errors

When authentication fails, the API returns one of the following error responses:

<ResponseField name="401 Unauthorized" type="error">
  Returned when the `X-API-KEY` header is missing, the key has been revoked, or the key has expired. Check that you are sending the correct key and that it has not been deleted from your dashboard.

  ```json
  {
    "success": false,
    "error": {
      "code": "UNAUTHORIZED",
      "message": "Invalid or missing API key"
    }
  }
  ```
</ResponseField>

<ResponseField name="403 Forbidden" type="error">
  Returned when your key is valid but the associated courier account does not have permission to access the requested resource. Verify that your key belongs to a courier with the required access level.

  ```json
  {
    "success": false,
    "error": {
      "code": "FORBIDDEN",
      "message": "Your API key does not have permission to access this resource"
    }
  }
  ```
</ResponseField>

## Security Best Practices

API keys are long-lived credentials that carry significant access. Follow these practices to keep your integration secure.

<Steps>
  <Step title="Store keys in environment variables">
    Never hard-code an API key directly in your source code. Store keys in environment variables or a secrets manager (e.g., AWS Secrets Manager, HashiCorp Vault, Doppler) and read them at runtime.
  </Step>
  <Step title="Never expose keys in frontend or client-side code">
    API keys must only ever be used from server-side environments. If a key is bundled into a mobile app, browser script, or any client-accessible artifact, it can be extracted and abused by third parties.
  </Step>
  <Step title="Rotate keys regularly">
    Generate a new API key and retire the old one at least every **90 days**. Shorter rotation cycles are recommended for high-volume or high-sensitivity integrations.
  </Step>
  <Step title="Use separate keys for each integration">
    Assign a unique API key to every distinct integration, service, or environment (development, staging, production). This limits blast radius if one key is compromised and makes it easy to audit or revoke access per integration.
  </Step>
</Steps>

<Warning>
  API keys grant **full courier-level access** to all packages, shipments, and settings associated with that courier account. Treat them with the same level of care as passwords — never share them in chat, email, or public repositories.
</Warning>

<Tip>
  For step-by-step instructions on rotating an existing key without downtime, see the [API Key management guide](/integrations/api-keys).
</Tip>
